地理计算模型及应用 #

1.地理学、遥感、GIS集成的概念框架 #

  • 如果说地图是地理学的第二代语言，那么==地理信息系统==就是地理学的第三代语言（地理学思维的建立 场景的创建 不是地图）
  • 文字 －》影像 －》场景

2.GIS定义：景观的数字描述 #

要素1：可视的或可感觉的地球（传感器） 要素2：垂直的观察方法（摄影测量 + 遥感） 要素3：二维的区域差异 要素4：作用机理，循环和过程（==该点当下GIS没有实现==） 最主要的描述 和 分析手段是：数量过程模拟。指责GIS没有“分析”，很大原因归咎于GIS缺乏这类过程模型，仅是一种“艺术”形式的系统 要素5：多种地理现象的集成（==该点也没有实现==） 依赖于数量模型的空间关系 和 空间过程分析模型 还没有很好地集成到GIS的框架中。 要素6：文化（人类的文明，人文景观）（==该点也没有实现==） 文化本身描述目前不充分，需要劳动密集型的采集数据，更需要GIS提供分析方法（人工智能、模拟思维、推演） 要素7：三维（==该点也没有实现==） 地表过程实际发生的行为空间要求景观模型的描述是空间完成的。 目前商业软件没有三维几何 和 拓扑。 要素8：时间变化（==该点也没有实现==） 最终解决办法不仅是一个时钟，而是必须处理空间以及随时间变化的问题。 （动态行为模拟+虚拟地理环境，全息地图） 要素9：景观单元之间的空间相互作用（==该点也没有实现==） 传输模型、过程模型 和 GIS的功能必须紧密地结合起来（陆面过程模型，地球模拟器） 要素10：景观概念的一致性（==该点是1－9的集成，当然也没有实现==） 三维的、功能性单元

三、端口 #

端口代表计算机上一个程序的进程

端口概念及常识 #

  • 不同的进程有不同的端口号，用来区分软件
  • 端口号范围被规定 0~65535
  • 不同协议相同端口不冲突；单个协议下端口号不能冲突

端口分类 #

  • 共有端口 0~1023

    • HTTP : 80
    • HTTPS : 443
    • FTP : 21
    • SSH : 22
    • Telent : 23

  • 程序注册端口

    • Tomcat : 8080
    • MySql : 3306
    • Oracle : 1521
    • redis : 6379

  • 动态、私有端口

    bash
    netstat -ano  #查看所有的端口
netstat -ano|findstr "5900"  #查看指定的端口
tasklist|findstr "1896"  #查看指定端口的进程

补充：跨域问题 #

  • 协议，端口，主机三者相同即为同源，反之，其中只要某一个不一样则为不同源
  • 同源产生的问题
    • Cookie、LocalStorage 和 IndexDB 无法获取
    • DOM 无法获得
    • AJAX 请求不能发送
  • 本地开发解决跨域方式
    • vite 的 proxy 进行代理
    • 后台开启 cors
    • 使用 nginx 转发请求
  • 生产环境解决跨域方式
    • 后台开启 cors
    • 使用 nginx 转发请求

注意：后台开启cors不需要前端做任何改动、nginx更改配置

四、通信协议 #

网络通信协议：速率、传输码率、代码结构 和 传输控制等

TCP/IP协议簇

应用层：操作系统中具体的程序应用

传输层：网络通信（TCP/UDP）

网络层：数据封包过程

数据链路层：光纤、电缆或卫星传输

TCP（电话） #

  • 连接 稳定
  • 三次握手 四次挥手
  • 客户端 和 服务端
    • 传输完成，连接释放，效率低

UDP（短信） #

  • 不连接 不稳定
  • 客户端、服务端：没有明确的界限
    • 只负责发送，不负责接收与否
